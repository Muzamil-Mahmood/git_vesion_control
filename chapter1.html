Chapter 01 — Beginner

Introduction to Version Control

What version control is, why every developer needs it, and how to set up Git for the first time.

What is Version Control?

Version control is a system that records changes to files over time so you can recall specific versions later. Think of it as an unlimited "undo" button for your entire project.

In mobile development, where you're juggling Kotlin, Swift, XML layouts, Gradle configs, and Podfiles — version control keeps everything synchronized across your team.

Without it, you'd be copying folders named app_v2_FINAL_FINAL — and we've all been there.

Key Insight: Version control isn't just for teams. Even solo developers benefit from it — it tracks your progress, lets you experiment safely, and gives you a safety net when things go wrong.

Why Version Control Matters

Version control solves real problems that every developer faces:

  • Track every change — See who changed what, when, and why. No more guessing who broke the build.
  • Experiment safely — Create branches to try new features without risking your stable code.
  • Collaborate effectively — Multiple developers can work on the same project simultaneously without stepping on each other's toes.
  • Roll back mistakes — Accidentally deleted something? Revert to any previous version in seconds.
  • Ship with confidence — Know exactly what's in every release. Tag versions so you can reproduce any build.

Git vs Other Version Control Systems

While there are many version control systems, Git has become the industry standard. Here's how it compares:

Feature Git SVN Mercurial
Type Distributed Centralized Distributed
Speed Very fast (local operations) Slower (needs network) Fast
Branching Lightweight and fast Expensive (copies entire tree) Lightweight
Offline work Full functionality Limited Full functionality
Industry adoption 93% of developers Declining Niche

Why Git won: Git's distributed model means every developer has a full copy of the repository. You can commit, branch, and merge entirely offline. When you're ready, push your changes to a remote like GitHub.

Setting Up Git

Before you start, you need Git installed on your machine. Here's how to set it up for each platform:

macOS:

# Install via Homebrew
brew install git

Windows:

# Download from git-scm.com or use winget
winget install Git.Git

Linux (Ubuntu/Debian):

sudo apt update
sudo apt install git

After installation, configure your identity. This information is attached to every commit you make:

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Verify it worked
git config --list

Important: Use the same email address that's associated with your GitHub account. This links your commits to your profile.

Your First Repository

Let's create a new repository and make your first commit. Open your terminal and follow along:

# Create a new project directory
mkdir my-mobile-app
cd my-mobile-app

# Initialize a Git repository
git init

# Create your first file
echo "# My Mobile App" > README.md

# Stage the file for commit
git add README.md

# Make your first commit
git commit -m "Initial commit: add README"

Congratulations — you've just created your first Git repository and made your first commit! Every change you make from now on can be tracked and reverted.

What Just Happened?

Let's break down those commands:

  • git init — Creates a hidden .git folder that stores all your version history.
  • git add — Stages changes, telling Git "I want to include this in the next commit."
  • git commit -m "message" — Creates a snapshot of your staged changes with a descriptive message.

Git tracks changes in three states: your working directory (files you're editing), the staging area (changes ready to commit), and the repository (committed history). Understanding this flow is fundamental to using Git effectively.
