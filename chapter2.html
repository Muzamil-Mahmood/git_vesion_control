Chapter 02 — Beginner

Core Concepts

Every VCS works around a few core ideas. Once you understand these, Git makes sense.

The Three States of Git

Before diving into commands, understand that Git tracks your files in three states:

  • Working Directory — Your actual files on disk. This is where you edit code.
  • Staging Area — A preview of what will go into your next commit. You "stage" changes here.
  • Repository — The committed history. Once you commit, the snapshot is permanent.

The flow is: edit files → stage changes → commit. This two-step process (add then commit) lets you craft precise commits that group related changes together.

# The basic workflow
git add file.swift # stage changes
git commit -m "msg" # commit staged changes

Repository

A repository (or "repo") is your project's entire history — every file, every change, every decision your team made. It lives in a .git folder at the root of your project.

When you run git init, Git creates this folder. When you clone a repo from GitHub, you get a complete copy of the entire history — not just the latest files.

# Clone an existing repository
git clone https://github.com/you/app.git

# This creates an app/ folder with the full history

Commits

A commit is a snapshot of your project at a point in time. It's like saving a photo of your code — you can always go back to it.

Every commit has:

  • A unique 40-character hash (e.g., a3f7b2c...)
  • A message describing what changed
  • A pointer to the previous commit (its parent)
  • A snapshot of all tracked files
git commit -m "feat: add profile screen"

# Good commit messages explain WHY, not just WHAT
git commit -m "fix: resolve crash on low memory devices"
git commit -m "refactor: extract auth logic into separate module"

Tip: Write commit messages in imperative mood ("add feature" not "added feature"). This matches the style used by git log and makes history easier to read.

Branches

Branches let you work on features in isolation. Your main code stays stable while you experiment on a separate copy.

Think of branches as parallel universes — you can create one, make changes, and decide later whether to merge it back into the main timeline.

# Create and switch to a new branch
git checkout -b feature/dark-mode

# Now you're on feature/dark-mode
# Make changes, commit, then switch back
git checkout main

# Merge the feature branch
git merge feature/dark-mode

Merging

Merging combines changes from different branches. It's how your team brings features together into the main app.

When you merge, Git tries to automatically combine the changes. If both branches modified the same lines, you'll get a merge conflict — Git will ask you to manually choose which version to keep.

# Merge a feature branch into main
git checkout main
git merge feature/dark-mode

# If there's a conflict, Git marks the files
# Edit the files to resolve, then:
git add .
git commit -m "Merge feature/dark-mode"

Merge Conflicts: Don't panic. Open the conflicted files, look for <<<<<<< markers, choose the correct code, remove the markers, and commit. It's a normal part of collaborative development.

Remotes

A remote is a version of your repo hosted elsewhere — like GitHub. It's how your team shares code across devices.

The default remote is called origin. When you clone a repo, Git automatically sets up the original as your origin remote.

# Add a remote
git remote add origin https://github.com/you/app.git

# Push your main branch to GitHub
git push -u origin main

# The -u sets up tracking, so future pushes are just:
git push

Pull Requests

A pull request (PR) proposes changes and lets your team review code before merging. It's quality control for your app.

On GitHub, you create a PR from your feature branch. Team members can review the code, leave comments, and approve or request changes. Once approved, you merge it into main.

# Create a PR using GitHub CLI
gh pr create --title "Add dark mode" --body "Implements theme switching"

# Merge after approval
gh pr merge --squash

HEAD and References

Git uses references (refs) to point to specific commits. The most important one is HEAD — it points to the commit you're currently looking at.

When you're on the main branch, HEAD points to the latest commit on main. When you switch branches, HEAD moves to the latest commit on that branch.

# See what HEAD points to
git log --oneline -1
a3f7b2c (HEAD -> main) feat: add login screen

# HEAD~1 means one commit before HEAD
git log --oneline HEAD~1

Tags

Tags mark specific commits as important — typically releases. Unlike branches, tags don't move. They're permanent markers.

# Create an annotated tag
git tag -a v1.0.0 -m "First release"

# List all tags
git tag

# Push tags to GitHub
git push origin --tags
