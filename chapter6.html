Chapter 06 — Advanced

Team Strategies & CI/CD

Choose the right branching strategy for your team and set up automated testing and deployment.

GitFlow Workflow

GitFlow is a structured branching model designed for projects with scheduled releases — perfect for mobile apps that ship to app stores on a fixed cadence.

Branch structure:

  • main — Production-ready code. Only release and hotfix branches merge here.
  • develop — Integration branch. All features merge here first.
  • feature/* — Feature branches. Branch from develop, merge back when done.
  • release/* — Release preparation. Bug fixes only, then merge to main AND develop.
  • hotfix/* — Emergency fixes. Branch from main, merge to both main and develop.

Best for: Mobile teams with versioned releases (v1.0, v1.1, v2.0). When you need release candidates, beta testing, and hotfixes for live apps.

GitHub Flow

GitHub Flow is simpler — just main and feature branches. Every change goes through a PR, gets reviewed, and merges to main. Main is always deployable.

The workflow:

  • Create a branch from main
  • Make changes and commit
  • Open a PR and get review
  • Merge to main after approval
  • Deploy from main

Best for: Teams practicing continuous deployment. When you can ship to production multiple times per day. Works well for web apps and teams with mature CI/CD.

Trunk-Based Development

Trunk-based development means everyone commits to a single branch (main/trunk) with short-lived feature branches (less than a day). Feature flags control what's visible to users.

  • Short-lived branches (hours, not weeks)
  • Feature flags to hide incomplete features
  • Continuous integration — merge to main at least daily
  • Automated testing catches regressions

Best for: Mature teams with strong testing culture. Google, Facebook, and Microsoft use variations of this approach.

Choosing a Strategy

Strategy Complexity Best For Release Cadence
GitFlow High Scheduled releases Weekly/Monthly
GitHub Flow Low Continuous deployment Multiple per day
Trunk-Based Medium Mature teams Continuous

Monorepo Strategies

A monorepo houses multiple projects in a single repository. Common in mobile development when you have an Android app, iOS app, and shared backend.

Tools for monorepos:

  • Nx — Smart monorepo tooling for JavaScript/TypeScript projects
  • Turborepo — High-performance build system for monorepos
  • Bazel — Google's build system for massive monorepos
  • Gradle (multi-project) — Native Android monorepo support

Challenge: Monorepos require careful CI configuration. Only rebuild/test projects that actually changed. Without this, CI times explode as the repo grows.

CI/CD for Mobile

Continuous Integration and Continuous Deployment automate building, testing, and releasing your app. Here's a basic GitHub Actions workflow for Android:

# .github/workflows/android.yml
name: Android CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-java@v4
      with:
        java-version: '17'
        distribution: 'temurin'
    - run: ./gradlew test
    - run: ./gradlew assembleDebug

Key CI/CD concepts for mobile:

  • Build verification — Ensure the project compiles on every push
  • Automated testing — Run unit tests, UI tests, and lint checks
  • Artifact generation — Build APK/IPA for testing
  • Deployment automation — Fastlane for app store submission
  • Code signing — Manage certificates and provisioning profiles securely
