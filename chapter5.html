Chapter 05 — Advanced

Advanced Git

Power-user techniques for cleaning up history, debugging with precision, and recovering from disasters.

Interactive Rebase

Interactive rebase lets you rewrite commit history — squash commits together, reorder them, edit messages, or split commits. It's the ultimate cleanup tool before opening a PR.

# Rebase the last 5 commits interactively
git rebase -i HEAD~5

This opens an editor with a list of commits. Each line starts with a command:

Command What It Does
pick Keep the commit as-is
squash Meld into the previous commit
fixup Like squash, but discard this commit's message
reword Keep the commit, but edit the message
edit Pause to amend the commit
drop Remove the commit entirely

Use Case: You made 8 small commits while debugging. Before opening a PR, squash them into 2-3 meaningful commits so reviewers can understand your changes at a glance.

Caution: Never rebase commits that have been pushed to a shared branch. Rewriting public history causes conflicts for everyone else on your team.

Cherry-Picking

Cherry-pick applies a specific commit from one branch to another. It's useful for backporting fixes or selectively applying changes.

# Apply a specific commit to your current branch
git cherry-pick a3f7b2c

# Cherry-pick multiple commits
git cherry-pick a3f7b2c b4c8d3e

# Cherry-pick without committing (stages changes only)
git cherry-pick --no-commit a3f7b2c

Use Case: You found a critical bug on the release branch. You already fixed it on the feature branch. Cherry-pick that fix directly to the release branch without merging the entire feature.

Git Bisect

Bisect uses binary search to find the exact commit that introduced a bug. Instead of checking commits one by one, it halves the search space each time.

# Start bisecting
git bisect start

# Mark current commit as bad
git bisect bad

# Mark a known good commit
git bisect good a1b2c3d

# Git checks out a middle commit
# Test it, then mark as good or bad
git bisect good # or git bisect bad

# Repeat until Git finds the culprit
# When done:
git bisect reset

Pro Tip: Automate bisect with a test script: git bisect run ./test.sh. Git will run the script on each commit and automatically mark good/bad.

Git Hooks

Hooks are scripts that run automatically at specific points in the Git workflow. They live in the .git/hooks/ directory.

Common hooks for mobile development:

Hook Runs When Common Use
pre-commit Before a commit is created Run linters, formatters, unit tests
commit-msg After commit message is entered Validate commit message format
post-merge After a merge completes Auto-install dependencies
pre-push Before push to remote Run integration tests
# Example: pre-commit hook for linting
# .git/hooks/pre-commit
#!/bin/sh
./gradlew lint
if [ $? -ne 0 ]; then
  echo "Lint failed. Commit aborted."
  exit 1
fi

Advanced .gitignore Patterns

Mastering .gitignore patterns helps keep your repository clean and secure.

# Ignore all .class files
*.class

# Ignore all .log files
*.log

# Ignore build directories
build/
/build

# But keep this specific build file
!build/config.gradle

# Ignore files only in root directory
/debug.log

# Ignore everything in temp/ except .gitkeep
temp/*
!temp/.gitkeep

Security: Never commit API keys, passwords, or certificates. Use .gitignore and environment-specific config files. If you accidentally commit a secret, rotate it immediately — it's in your history forever.

Submodules

Submodules let you include one Git repository inside another. Use this for shared libraries or components across projects.

# Add a submodule
git submodule add https://github.com/user/shared-lib.git libs/shared-lib

# Clone a repo with submodules
git clone --recurse-submodules https://github.com/user/app.git

# Update submodules to latest
git submodule update --remote --merge

Alternative: For simpler dependency management, consider package managers (CocoaPods, Gradle dependencies) over submodules. Submodules add complexity and are easy to get wrong.

Git Reflog

Reflog is Git's safety net. It records every movement of HEAD, including commits that seem "lost" after a reset or rebase.

# View reflog
git reflog

# Output shows all HEAD movements
a1b2c3d HEAD@{0}: reset: moving to HEAD~3
f4e5d6c HEAD@{1}: commit: important feature
g7h8i9j HEAD@{2}: commit: another change

# Recover a "lost" commit
git checkout f4e5d6c

# Or create a branch at that commit
git branch recover-branch f4e5d6c

Key Insight: Even after git reset --hard, your commits aren't truly deleted. They're just unreachable. Reflog gives you 30 days to recover them.
