Chapter 03 — Intermediate

Workflow & Mobile Dev

The standard Git workflow for mobile teams, plus platform-specific guidance for Android, iOS, and cross-platform development.

The Standard Git Workflow

Here's the workflow that most mobile teams follow. It keeps your main branch stable while enabling parallel development.

01

Create a Branch

Start every new feature or bug fix on its own branch. This keeps the main branch stable and deployable at all times.

git checkout -b feature/push-notifications
02

Build & Commit Often

Make small, focused commits. Each commit should represent one logical change — "add Firebase config" not "fix everything".

git add . && git commit -m "feat: configure FCM"
03

Push & Open a PR

Push your branch to GitHub and open a pull request. Describe what you built and why — your future self will thank you.

git push origin feature/push-notifications
04

Review & Merge

Your team reviews the code, runs CI tests (build the APK/IPA), and merges when approved. Automated checks catch issues early.

gh pr merge --squash
05

Deploy & Tag

Tag your releases so you can always identify which commit shipped to the App Store or Play Store.

git tag -a v1.2.0 -m "Release: push notifications"

Branch Naming Conventions

Consistent branch names make it easy to understand what each branch is for. Here are common patterns:

  • feature/dark-mode — New features
  • bugfix/crash-on-login — Bug fixes
  • hotfix/critical-security-patch — Urgent fixes for production
  • release/1.2.0 — Release preparation
  • chore/update-dependencies — Maintenance tasks

Writing Good Commit Messages

Good commit messages make your project history searchable and understandable. Follow the Conventional Commits format:

# Format: type(scope): description

feat(auth): add Google Sign-In
fix(playback): resolve audio stuttering on Android 12
refactor(profile): extract image loading to ViewModel
docs(readme): update installation instructions
test(auth): add unit tests for login flow

Common types: feat (new feature), fix (bug fix), refactor (code restructuring), docs (documentation), test (tests), chore (maintenance).

Android Projects

Git works seamlessly with Android Studio, Gradle, and the Android ecosystem. Here's how to manage your Android-specific files:

  • Track build.gradle changes across versions — dependency updates are critical history
  • Manage different flavors (dev, staging, prod) with separate config files
  • Handle google-services.json safely — never commit API keys
  • Separate SDK configs per environment
# Essential Android .gitignore entries
*.iml
.gradle/
.idea/
local.properties
/build
google-services.json

iOS Projects

Xcode projects have complex file structures. Git tracks your Swift code, storyboards, and CocoaPods/SPM dependencies.

  • Track .xcodeproj and .xcworkspace — but be careful with merge conflicts
  • Manage CocoaPods with Podfile.lock committed
  • Handle provisioning profiles via Fastlane
  • Use .gitignore for DerivedData and build artifacts
# Essential iOS .gitignore entries
xcuserdata/
*.xcworkspace/xcuserdata/
DerivedData/
build/
Pods/

Cross-Platform Projects

React Native, Flutter, and Kotlin Multiplatform projects need smart .gitignore files and CI pipelines for both platforms.

  • Manage node_modules (React Native) and pubspec.lock (Flutter)
  • Set up CI builds for iOS and Android simultaneously
  • Separate shared code from platform-specific modules
  • Use Fastlane for automated deployments to both stores

Handling Merge Conflicts in Mobile Projects

Merge conflicts are common in mobile projects, especially with Xcode project files and Gradle configs. Here's how to handle them:

Xcode Projects: The .pbxproj file is notoriously difficult to merge manually. Consider using a tool like xcodeproj or coordinating with your team to avoid simultaneous edits to project settings.

Best Practice: When resolving conflicts in build.gradle or Podfile, always test the build after merging. Conflicts in dependency declarations can cause silent failures.
